ul(unordered list) : 순서가 없는 리스트

  • ul 다음에 사용할 수 있는 태그는 li 밖에 없음
  • li 태그 다음에 사용할 수 있는 태그는 li 밖에 없음
  • ul은 li태그들의 부모역할만 담당함

ol(ordered list) : 순서가 있는 리스트

블릿기호

순서가 없는 리스트의 경우, 앞에 점이 붙어있음

순서가 있는 리스트의 경우, 앞에 숫자가 붙어있음

리스트 앞에 점, 숫자, 사각형, 다른 기호들이 붙어있을경우 이를 "블릿기호"라고 한다.

ol list 활용하기

  1. ul처럼 ol리스트를 사용할때는 반드시 li태그가 우선된다.
  2. ul과 마찬가지로 ol은 li밖에 담을 수 없음
  3. li 다음에 사용할 수 있는 태그는 li로 사용방법이 동일하다.
  1. ol이 작성되고 li태그 사용하면 무조건 1번부터 시작
  2. list는 하나만 사용하는것이 아니라 여러장의 페이지를 구성하려고 작성함
  3. list를 다룬다면 2개 이상의 비슷한 영역이 필요함
  1. ul li를 사용할경우, 앞에 점이 붙어있음
  2. ol li를 사용할경우, 앞에 숫자가 붙어있음
  3. 기호나 숫자는 사용자가 원하는대로 바꿔서 사용이 가능하다.
  4. 앞에 붙어있는 기호들을 "블릿기호" 라고 한다.

overflow 기능들

  1. overflow 기능들 중, hidden을 제일 많이 사용함
    (넘치는 객체, 넘치는 내용, 넘치는 이미지, 넘치는 영역)
  2. overflow 기능들 중, scroll
    웹은 정보 전달의 목적도 있기때문에 읽을 수 있어야한다.
  3. overflow 기능들 중, auto
    scroll이 넘치는 방향으로만 보이도록 만들기
  4. 실제 면접에 나왔던 문제
    :: auto와 scroll중 많이 사용하는것은? (디자인 부서에서 나온 문제)
    :: auto는 어느 방향으로 늘어날지 알 수 없기때문에 미리 대비하기 어려움
    :: scroll은 사용하지않아도되는 scroll을 표현하지만 어느 정도 공간을 사용할지
    알 수 있기때문에 정확한 픽셀을 알 수 있음

(웹 분야) 구분하기

  1. 웹 디자인 / 디자인 + 퍼블리셔(디퍼블리셔)
  2. 개발자 + 퍼블리셔 (개퍼블리셔)
  3. 프론트엔드
  4. 프론트엔드, 프론트엔드 + 디자인
  5. 프론트엔드, 프론트엔드 + script 개발자
  6. 서버사이드 (.js(서버에 맞춰서 사용하는 script들), 노드, 앵귤러, 리액트)
  7. 프론트엔드 + script + database (세미풀스텍)
  8. 백엔드 (php, jsp, asp)
  9. 풀스텍 개발자 : 웹 퍼블리셔 + 백엔드 (php, jsp, asp) + 보안

웹 순서 나누기

  1. 자료수집(이미지 촬용, 스케치, 비율에 맞는 이미지)
  2. 관련 사이트 조사(url,기획,구성)
  3. 멘트작성
  4. [마크업] wrap, header, container (section 몇 개?, side), footer 영역 나누기
    해당하는 영역에서 어떤 태그를 사용할것인지 이미지 비율은 어떻게 될 것인지 구상하는 단계
  5. [header] 영역 설계
    + 상징적인 요소들 h1 || div(id="logo", class="logo")
    + 로고를 만들때 : 로고 이미지, 상징적인 폰트, 대표 상품이미지
    + 네비게이션 제작: nav, gnb, ul li, ol li 태그들 구성 확인
    + 기능은? "세로로 한줄씩 떨어지는 기능", "한줄이 가로로 길게 나오는 기능", "전부 보여주는 기능"
  6. [container] 영역 설계
    + section 은 몇 개로 구분할지 설계
    + side 메뉴 설계?? side가 있는 페이지와 없는 페이지가 있음
    + visual 영역은 한 장면에 2단? 3단 어떤 구조로 만들지 설계
    + visual 영역의 기능? (동영상, html5를 이용한 애니메이션, script를 이용한 슬라이딩 윈도우 구현)
  7. [footer] 영역 설계
    카피라이터, 제휴업체(스폰서), 2중 구조, 3중 구조
    footer영역에 로고 제공
  8. [wrap] 영역 설계
    전체를 사용하는 영역과 실제 사용영역으로 구분하기
    2016년 960px, 1025px, wrap에 의존하는 페이지들이 많았음
    현재의 페이지들은 각각 영역에 의존하는 페이지가 많음
    : 전체 사용하는 영역 width:100%를 구성하여 "와이드"하게 사용
    : 실제 사용하는 영역 px로 구분되는 영역을 구성하여 가운데 배치함
    매우중요) 마우스로만 되는것이 아니라 키보드로도 조작이 가능해야 함

웹 페이지 설계(실무 기준)

  1. xhtml 페이지 만들기(웹 표준, 웹 접근성을 고려하는 "정적인 페이지" 제작)
    + 정적인 페이지: 링크요소 설정, 동적인 기능은 제공하지 않음
    + css3, script: 애니메이션, 동영상 기능, 움직이는 기능, 애니메이트
  2. xhtml 페이지 제작 완료 + script로 기능만들기
    xhtml(정적인 페이지) + jQuery, javascript(동적인 페이지) 제작
    : html5는 제외!
    문제점: script 영역에 대한 의존도가 매우 높아서 기능이 매우 느리게 표현되는 문제가 발생함
    페이지를 읽어주는 속도는 빠르지만 기능은 매우 느림
  3. html5(페이지 만들기) "표준 브라우저 전용으로 만들기 시작함" html5는 웬만해서는 오류가 발생하지 않음
    문법이 틀리고있지만 고장이 나지 않아서 모바일이나 태블릿pc에서 쉽게 고장이 발생함
  4. html5로만 만들 경우 "css3를 받아들이지 못하는 익스플로러에서는 최하의 성능을 보임"
  5. html5의 빠른 css3 효과와 script의 기능을 함께 섞어서 사용하면 느려지는 속도를
    보완할 수 있음 (xhtml에 대한 기초가 중요함)
  6. *css3에서 배우는 동적인 요소들
    (트랜스폼, 장면전환, 멀티기능,애니메이션,반응형, 절대형, 적응형)
  7. html5 필수! (유효성 검사, 로그인, 회원가입)
    + 유효성 검사(아이디 입력, 비밀번호 입력, 필수 입력사항 반드시 기입)
  8. html5 필수_2 (반응형, 절대형,적응형) + script 만들기
  9. [정리] html5의 구조에 맞춰서 페이지를 제작하고,
    javascript + jQuery 로만 기능을 만들던것을 서로 보완하여 사용함

    :: css3와 script를 섞어서 기능이나 웹 페이지를 만들어야 속도를 보완할 수 있음
