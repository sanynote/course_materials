모든 영역, 모든 객체를 div 태그로 만들어서 제작함(네이버(현재))
div태그를 사용하면 어떤 목적으로 작성이 된 영역인지 확인이 필요함!
이때 등장한 값이 목차 생성(headline 태그 작성 권장)
div태그를 사용하고, 목차를 만들지 않는 경우가 많아져서 div태그에 대한
이해도가 매우 떨어지는 문제 발생!
* 시멘틱태그는 "이유가 있는 태그" 를 만들고, div태그에 대한 의존도 낮추기가 핵심!
* 시멘틱태그를 사용하면서 어떤 목적으로 만들었는지 목차를 만드는 것이 중요한
역할이 되어버림

* 목차생성 페이지 : outliner (html5, 메인페이지)

시멘틱 태그란?

  1. 무분별하게 사용되는 div태그의 사용용도 낮추기
  2. * 약속된 작업을 하기위한 "이유있는 태그!"
  3. div, p 태그의 경우 제목이 없다면 왜 사용되었는지 알기 어려움!
  4. header, footer, section, nav 등만 봐도 뭐를 해야 할지 알 수 있는 태그들이 시멘틱 태그!

header태그 || header 영역

  3. html5, 반응형 (부트스트랩 , 자동으로 만들어진 템플릿)
    사용자가 직접 id나 class명을 기입하기 어려움!
    이유는? 자동화 도구로 만들어져 있어서 교체하거나 변경하는 작업이 어려움!
    [실제 생겼던 문제점]
    : header 영역안에 nav태그를 사용할 수 없는 문제가 발생함!
    : a태그에 링크가 걸리지 않는 문제가 발생함
    (header 영역안에 기본적으로 nav를 못 받아주는 문제발생)
    : 자동화에 성립되지 않는 문법은 고장날 확률이 매우 높음!
    해결1: header영역 바깥에 nav영역을 따로 만들어서 해결함!
    해결2: header태그를 div로 변환하면 해결이 가능했지만, 자동화가
    깨지는 문제가 발생함!
    * 자동화 도구는 언어바뀌거나, 문법이 달라지면 자동화가 쉽게 깨지기
    때문에 구조를 보고 직접 작성하는 것을 권장한다!

중요사항

1) header, footer 영역은 한 페이지에서 한번밖에 사용하지 못함!
2) header태그를 2번 사용하는 것은 머리 2개 만드는 것과 같음
3) footer태그가 늘어나는 것은 다리갯수를 늘리는 것과 같음
4) header와 footer 영역안에는 아주 특별한 이유가 없다면 section태그를 사용하지 않는다!
5) section 태그는 주로 테마나 주제에 맞춰서 영역 구분할때 사용하기 때문에 container에서 자주 사용된다.

section 영역에 대한 이해

  1. container영역은 다양한 주제를 다룰때 사용하는 영역
  2. visual영역은 주로 광고 목적으로 사용하기 때문에 section처럼 연결해서 사용하지 않음(독립요소)
  3. visual은 마디역할로 구분하기 보다 하나의 주제를 다루는 독립요소로 div태그를 주로 사용한다!
  4. * 전체 사용 영역을 주제별로 나열할때 section 태그 사용!
  5. * 비슷한 정보의 값들을 모아둔 영역이기 때문에 제목을 보고도 어떤 작업을 하는지 알 수 있어야 함!
  6. * section영역안에 section 태그 사용가능
  7. 큰 주제로 묶어주고, 그안에서 비슷한 정보를 세분화여 보관할때 section안에 다시 section태그 사용가능함!
  8. * h1은 사용은 할 수 있으나.. header에서 주로 사용하는 중요도가 높은 태그가 h1
  9. section은 비슷한 주제를 계속 연결하는 개념으로 "형제 요소처럼 나열" 되는 경우가 많은데
    h1을 어느 영역에 사용하게 되면 해당하는 영역만 중요도가 높아져서 문제가 될 수 있다!

html5문서의 특징

  1. 기본 원칙, 한 줄짜리 태그는 끝나는 부분에 "/" 사용 필수
  2. html5 문서는 한 줄짜리 태그에 닫는 "/" 가 없어도 가끔 고장나지 않는 경우가 있음
  3. validator에서 오류로 잡지 않을때가 있음! html5는 매우 융통성이 좋음!
  4. 안드로이드, 아이폰, 윈도우, mac , 운영체제가 모두 다름!
  5. * html5가 뛰어나지만 모든 운영체제를 받아들이기는 어려움!
  6. 매우중요) 웹 점검도구나 디바이스기기들에서 "/" 닫는 부분이 없다면 감점요인이 되거나 기능이 고장나는 문제가 될 수 있기 때문에 필수 작성되어야 한다!
  7. * html5는 주로 문제점을 error 보다는 warning이 자주 발생함!
  8. * html5에서 에러가 발생한다면 반드시 수정하고 고쳐야 디바이스 기기들에 문제가 발생하지 않는다
  9. 중요) html5 문서로 작성하다 보면 validator 작업시 warning 이라는 경고가 뜨지만...
    실제로 보이지 않는 경우가 발생함!
  10. validator에서 몇 번째 줄에서 에러가 발생했는지 알 수 없는 경우가 발생하는데..
    이유는? html5는 광범위한 작업이 가능하기 때문에 "위험할 수 있다" 라는 경고를 보여주는 부분!
  11. 정확하게 어디에서 에러가 발생했는지 , 위험요소가 어디에서 나타났는지 알 수 없을 경우,
    무시하고 작성해도 상관없음
  12. 정리) html5는 스스로 에러나 오류를 어느정도 해결 할 수 있지만, 에러나 오류가 보인다면 치명적인
    부분으로 다가올 수 있기 때문에 반드시 수정해야 한다!

xhtml vs html5 버전의 차이 (style 문제풀기)

  1. xhtml 에서는 validator를 확인하면 style에 type="text/css" 작성하기!
  2. html5 에서는 type="text/css"를 작성하면 에러가 발생함!
  3. + xhtml에서 사용하는 css 버전은? css1 || css1.0 || css 버전
  4. + xhtml에서 사용하는 문서의 버전은? xml1.0 || xml1
    + html 문서 버전은 4.0 || 4
    + xml + html 문서 합쳐서 5.0이 아니라 transition.dtd 문서를 사용하여
    호환하도록 만듬!
  5. + html5에서 사용하는 css 버전은? css3 || css3.0 || css3+

  6. xhtml에서는 type="text/css" 작성했지만, html5에서 작성하지 않는 이유는?

    작성을 할 경우, 에러가 발생함! (2018년도 하반기 까지는 작성하는 것을 권장!)

    html5는 스스로 장착하고 있는 요소가 많고, 광범위하게 사용되는 문서로 미리 저장되어
    있는 값들이 많이 있음!

    html5는 보이지 않지만 type="text/css"를 장착하고 있기 때문에 type="text/css"를
    작성하면 2번 작성한 것과 같음! (에러가 발생함)

  7. * html5와 xhtml은 독타입 (선언문) 부터 다르기 때문에 작성되는 형태도 달라질 수 있다
    * 독타입을 확인하고 적절한 버전의 값을 작성해야 한다!

인식률

div, p, ul li, ol li, a, img 등등의 태그들은 초반부터 만들어진 태그들로 웹에서 사용되는 인식률이 매우 높음!
시멘틱태그는 처음부터 만들어진 태그가 아니라 목적에 의하여 나중에 만들어진 태그들로,
웹에서 인식률이 떨어지는 문제가 발생함! (class 적용했으나 적용되지 않는 느낌과 비슷함)

매우중요

시멘틱 태그들을 사용할 경우, 인식률이 저하되는 원인을 막기 위하여 id나 class를 사용한다!
style에서도 태그로 호출하지 않고, #header || .header
id나 class로 호출함!
* 인식률이 떨어질 때는 css는 물론 script 영역에서도 문제가 되어 기능이 적용되지 않는다!

html5 문서의 사용목적

html5를 사용하는 이유는? 모든 디바이스들과 호환이 가능한 확장성을 가지고 있음

xhtml은 버전의 한계가 있으나 html5는 한계가 없음!

* html5의 css 버전은? css3 || css3.0 || css3++

css3의 장점은 동적인 기능을 많이 가지고 있음 (움직이는 효과를 만들 수 있음)
중요) script의 느려지는 부분을 css3로 보완할 수 있음!

단점: 호환성이 뛰어나기 때문에 문법을 html5로만 사용하게 되면 쉽게 고장남
: 구형브라우저는 고려하지 않음!

장점: 표즌 브라우저를 기준으로 css3의 모든 장점을 사용할 수 있음!
script와 호환성이 매우 뛰어남

section id="best" / section id="sale"

tip) 시멘틱 태그와 id를 사용했음에도 문제가 발생할 경우 (거의 없는 문제)
